The honest summary

Rondo is a well-built app. The code is clean, the features are real, and it handles money, real-time data, and user auth properly. It is not a toy project — it has tournament brackets, wallet payments, social feeds, and live game tracking all working together.

Before the fixes, the main problems were things users feel but might not be able to name:

  • Pages jumped around while photos loaded (annoying)
  • The server was doing the same work over and over on every tap (wasteful)
  • Starting a tournament made you wait 1–3 seconds staring at a spinning button (slow)
  • Sending notifications was holding up the server response (unnecessary)

All four of those are now fixed.

What Got Fixed (plain English)

Fix 1 — UX

Pages stopped jumping when images load

When you opened any screen with photos, the layout would shift and jump around while the images downloaded. This is called Cumulative Layout Shift — a real metric browsers measure. It happens when the app doesn't tell the browser how big a photo is before it arrives, so the browser guesses wrong and has to rearrange everything once the image shows up.

Before
Text and buttons shifted down when profile photos, game covers, and post images loaded. Felt glitchy.
After
The browser reserves the exact right amount of space before the image arrives. Nothing moves.
Fix 2 — Server

The server stopped redoing the same work on every request

Every time someone tapped anything, the server was rebuilding the same things from scratch: reconnecting to the database, re-encoding a PayMongo password in Base64, and rebuilding a list of banned words into searchable patterns. All of that is constant — it never changes — yet the server was doing it thousands of times per day.

Before
Database connection, PayMongo credentials, and word filters rebuilt on every single API call.
After
Built once, stored in memory, reused forever. Same result, far less work.
Fix 3 — UX

More image layout shift fixes (second pass)

After fixing the obvious images, a second sweep found more places where photos lacked reserved space. The most notable one was post media in the social feed — an image set to 800×450 pixels on paper but with "height: auto" in the style, meaning the browser still had to guess until the real image arrived.

Before
Social feed posts, profile avatars, and game cover images all caused subtle layout jumps.
After
Every image now has a reserved container. The page is stable from the first render.
Fix 4 — Speed

Starting a tournament no longer freezes the button for seconds

When an organizer clicked "Start tournament", the server had to generate the full fixture bracket, update every team's seeding, save all the matches to the database, update the tournament status, and send notifications to every team captain — all before it would say "done" and unfreeze the UI. For a 16-team tournament that's nearly 50 database operations and could take 1–3 seconds.

Now the server just validates the request, marks the tournament as "Generating…", drops the job in a queue, and immediately responds. The actual bracket work happens in the background. The page updates automatically when it's ready via a live database connection.

Before
Organizer clicks "Start" → spinner for 1–3 seconds → page unfreezes. If something went wrong, silent failure.
After
Organizer clicks "Start" → instant "Generating…" message → bracket appears on its own. Retries automatically if it fails.
Fix 5 — Performance

Notifications stopped slowing down every action

Across the app, every meaningful action (liking a post, commenting, paying for a game, approving a player, adding from the waitlist, recording a match result) sent a notification before the server would reply. The notification is a nice-to-have — the user doesn't need to wait for it. They needed their payment confirmed, not to wait for a ping to be stored.

All seven of those notification inserts now run after the response is sent. The user gets their confirmation instantly and the notification saves itself in the background.

Before
Pay for game → wait for payment + notification → see confirmation. Every action had a hidden 50–100ms tax.
After
Pay for game → see confirmation immediately. Notification saves itself silently afterwards.

What is still good (unchanged)

  • Security is solid. Payment anti-fraud checks, rate limiting on comments and posts, Row Level Security on every table, guest-user guards on every write operation — all already in place and untouched.
  • The data model is clean. Tournaments, teams, matches, wallets, notifications, and profiles are all properly related. No shortcuts that would cause problems later.
  • Real-time features work well. The live bracket updates, timer sessions, and chat messages all use Supabase channels correctly.
  • PayMongo payments are robust. The idempotency keys, anti-fraud ledger, and webhook verification were already well-implemented.
